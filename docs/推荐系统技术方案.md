# MetadataTJ 推荐系统技术方案

## 一、系统概述

MetadataTJ 是一个基于语义理解的数据元/限定词智能推荐系统，用于将地方库字段自动对标到公安标准数据元。系统采用 **三阶段流水线** 架构，结合向量检索、LLM语义理解和历史推荐优化，实现高精度的自动对标。

### 核心能力

| 能力 | 说明 |
|------|------|
| 数据元推荐 | 根据字段中文名/英文名，推荐最匹配的标准数据元 |
| 限定词推荐 | 识别字段中的修饰词，推荐对应的标准限定词 |
| 表映射 | 地方库表/字段映射到标准库表/字段 |
| 历史优先 | 基于历史对标记录，优先返回最佳匹配结果 |

## 三、三阶段流水线详解

### 阶段1：字段语义分解

**目标**：将来源字段分解为「限定词 + 数据元核心」

**实现方式**：
- **规则分解**（默认）：根据后缀（姓名、地址、日期等）提取核心语义
- **LLM分解**（可选）：调用大语言模型进行语义分析

**示例**：
```
输入: "父亲姓名"
输出: {
  "core_element_hint": "姓名",
  "qualifier_hints": ["父亲"],
  "confidence": 0.8
}
```

**代码位置**：[src/llm/decompose.py](file:///d:/workspace/metadataTJ/src/llm/decompose.py)

---

### 阶段2：向量召回（粗排）

**目标**：从向量库中召回语义相似的候选数据元

**技术方案**：
- **向量库**：ChromaDB（嵌入式，支持本地持久化）
- **Embedding**：调用外部 Embedding API（如 DashScope）
- **相似度计算**：余弦相似度
- **召回数量**：Top 50（可配置）

**索引构建**：
```bash
python scripts/build_index.py
```

**代码位置**：[src/vector/chroma_store.py](file:///d:/workspace/metadataTJ/src/vector/chroma_store.py)

---

### 阶段3：LLM精排（重排）

**目标**：对召回的候选结果进行语义精排，选出最匹配的 Top N

**实现方式**：
- 构建候选列表提示词
- 调用 LLM 进行语义排序
- 返回带分数和理由的排序结果

**Prompt 示例**：
```
你是公安标准数据元对标专家。根据来源字段，从候选数据元列表中选出最匹配的 Top N。

来源字段：父亲姓名 (father_name)
分解提示：姓名
请选出 Top 5 数据元。

候选列表：
- DE001 | 姓名 | name | 召回分=0.95
- DE002 | 人员姓名 | person_name | 召回分=0.88
- DE003 | 关系人姓名 | relation_name | 召回分=0.82
...
```

**代码位置**：[src/llm/rerank.py](file:///d:/workspace/metadataTJ/src/llm/rerank.py)

## 四、历史推荐优化

### 4.1 设计目标

基于已有的历史对标记录，统计每个来源字段对应各数据元的匹配次数，优先返回匹配次数最多的结果，提升推荐准确率。

### 4.2 架构设计

```
┌─────────────────────────────────────────────────────────────────────────────┐
│                           历史推荐数据流                                     │
├─────────────────────────────────────────────────────────────────────────────┤
│                                                                             │
│  ┌─────────────────┐                                                        │
│  │  历史对标记录表   │  已有的历史对标数据                                     │
│  │  (rucp_element_  │  记录: source_cname → target_element_code            │
│  │   mapping_       │  由其他系统/人工产生                                    │
│  │   history)       │                                                        │
│  └────────┬────────┘                                                        │
│           │                                                                 │
│           │ 定时同步（每24小时）                                              │
│           │ GROUP BY + COUNT(*) 统计匹配次数                                 │
│           ↓                                                                 │
│  ┌─────────────────┐                                                        │
│  │   统计表         │  存储匹配次数统计                                       │
│  │  (rucp_history_  │  source_cname → target_element_code → match_count    │
│  │   recommend_stat)│  match_count = 该组合出现的次数                        │
│  └────────┬────────┘                                                        │
│           │                                                                 │
│           │ 应用启动时加载                                                    │
│           ↓                                                                 │
│  ┌─────────────────┐                                                        │
│  │   内存缓存        │  快速查询，无IO开销                                     │
│  │  (dict)         │  key=source_cname, value=List[HistoryRecommend]       │
│  └─────────────────┘                                                        │
│                                                                             │
└─────────────────────────────────────────────────────────────────────────────┘
```

### 4.3 同步策略

| 同步类型 | 触发条件 | 说明 |
|---------|---------|------|
| 全量同步 | 首次启动 / force_full=true | 统计所有历史对标记录 |
| 增量同步 | 定时任务（每24小时） | 仅统计 id > last_sync_id 的记录 |

**增量同步原理**：
```sql
-- 查询新增的历史对标记录
SELECT source_cname, target_element_code, COUNT(*) as match_count
FROM rucp_element_mapping_history
WHERE id > {last_sync_id} AND status = 1
GROUP BY source_cname, target_element_code;

-- UPSERT 更新统计表
INSERT INTO rucp_history_recommend_stat (...) VALUES (...)
ON DUPLICATE KEY UPDATE match_count = match_count + VALUES(match_count);
```

### 4.4 推荐合并策略

```
推荐请求 → 查询历史缓存 → 合并候选 → 重排序 → 返回Top5
              ↓
         历史Top1优先（按match_count降序）
         剩余位置由向量召回补充
```

## 六、核心流程代码

### 6.1 推荐主流程

```python
def recommend(self, cname: str, ename: str, ...) -> dict:
    # 1. 字段语义分解
    decomp = decompose_field(cname, ename, self.llm)
    
    # 2. 构建检索查询
    queries = build_retrieval_queries(cname, ename, decomp)
    
    # 3. 向量召回
    raw_candidates = self.store.search_elements(queries["element"], top_k=50)
    
    # 4. 合并历史推荐（历史优先）
    candidates = self._history.merge_with_candidates(cname, ename, raw_candidates)
    
    # 5. LLM精排
    rankings = rerank_elements(cname, ename, decomp.core_element_hint, candidates, top_k=5)
    
    # 6. 返回结果
    return {"element": {...}, "deteminer": {...}}
```

### 6.2 历史推荐合并

```python
def merge_with_candidates(self, source_cname, candidates, top_k=5):
    # 1. 查询历史缓存
    history = self._cache.get(source_cname, [])
    
    # 2. 历史结果转候选格式
    history_candidates = [
        {"element_code": h.target_element_code, "score": 1.0, "is_history": True}
        for h in history
    ]
    
    # 3. 去重合并
    used_codes = {h.target_element_code for h in history}
    remaining = [c for c in candidates if c["element_code"] not in used_codes]
    
    # 4. 历史在前，候选补充
    return history_candidates[:top_k] + remaining[:top_k - len(history_candidates)]
```

LLM 调用通过 [src/llm/client.py](file:///d:/workspace/metadataTJ/src/llm/client.py) 封装，可替换为：

- 其他云服务

数据加载通过 [src/knowledge_loader.py](file:///d:/workspace/metadataTJ/src/knowledge_loader.py) 封装，支持：
